## 17. 风险管理

| 风险 | 影响 | 处理策略 |
| --- | --- | --- |
| MVP 范围扩大 | 延期 | 新需求先进入需求池 |
| Windows 环境缺失 | 无法打包 | 阶段 1 提前检查 |
| 提醒不稳定 | 用户错过任务 | 第一版限定应用运行时提醒 |
| 数据迁移不规范 | 升级后丢数据 | 阶段 2 建立迁移机制 |
| UI 打磨耗时 | 主流程延期 | 先完成可用界面，再局部优化 |
| 测试不足 | 发布后问题多 | 阶段 8 集中补齐主流程测试 |

## 20. V1.0 完成定义

Torder（今序）V1.0 完成时，应满足：

- Windows 桌面端可以安装
- 应用可以稳定启动
- 用户可以创建、编辑、删除、完成任务
- 今日任务、全部任务、已完成任务、过期任务视图可用
- 任务可以设置截止时间、提醒时间、优先级、清单和标签
- 搜索和筛选可用
- 本地数据库可持久保存数据
- 导入导出可用
- 主题切换可用
- 基础提醒可用
- 托盘基础能力可用
- 主要流程经过测试
- 安装包和发布说明已归档

## 21. 下一步建议

下一步建议从阶段 0 开始，不直接写代码。先用半天到一天确认原型和 MVP 边界，再进入阶段 1 初始化工程。

如果目标是最快做出能用版本，可以将阶段 0 控制在 2 到 4 小时，只确认四件事：

- 首页长什么样
- 任务有哪些字段
- 第一版哪些功能必须做
- 哪些功能明确不做

确认后即可启动开发。
